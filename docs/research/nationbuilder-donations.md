# NationBuilder Donations: "Number of Donations" and "Funds Raised" for a `/dashboard` year table

Research for the planned `/dashboard` page (part of the wayfinder effort tracked in issue #18; other metrics in that effort — e.g. volunteer signups — are being researched in parallel on sibling branches not yet merged to `main`). Question: how would a year-by-year "Number of Donations" and "Funds Raised" metric be computed from NationBuilder's v2 API, and what does the *donations* data model actually look like — as opposed to the `signups` endpoint this repo already calls (`SIGNUPS_API` in `src/util.tsx`, used by `src/ContactExport.tsx`)?

All claims below are sourced from NationBuilder's own documentation and its official machine-readable API spec:
- The v2 OpenAPI spec itself, downloaded from the "Download OpenAPI spec" link on the interactive reference page ([`https://nationbuilder.com/api/v2/reference`](https://nationbuilder.com/api/v2/reference) → [`https://nationbuilder.com/docs/v2/released.yaml`](https://nationbuilder.com/docs/v2/released.yaml), `openapi: 3.1.2`, `info.title: NationBuilder V2 API`, `info.version: '2.0'`). This is the ground-truth schema/paths reference and is cited as **"OpenAPI spec"** below.
- [NationBuilder v2 API Core Concepts](https://support.nationbuilder.com/en/articles/9757369-nationbuilder-v2-api-core-concepts) (HOWTO)
- [Relationships between Resources in v2 API](https://support.nationbuilder.com/en/articles/9912495-relationships-between-resources-in-v2-api) (HOWTO)
- [API v2 walkthrough](https://support.nationbuilder.com/en/articles/9899245-api-v2-walkthrough) (HOWTO)
- [Donations API](https://nationbuilder.com/donations_api) (legacy v1 marketing/reference page, used only as a pointer — the v1 endpoints it documents are **not** what the org would build against)
- [Recurring donation processing](https://support.nationbuilder.com/en/articles/2351041-recurring-donation-processing) (HOWTO, control-panel/UI behavior, not an API doc)

Nothing here was inferred from blog posts, forum posts, or Stack Overflow. Where a secondary search snippet surfaced a claim I could not confirm in the primary docs, I say so explicitly rather than repeating it as fact.

## TL;DR

- The endpoint is **`GET /api/v2/donations`** (JSON:API, same shape as `signups`). There is no dedicated donations summary/aggregate endpoint, and **no `recurring_donations` resource exists in the v2 API at all** — I grepped the full 640KB OpenAPI spec for "recurring" and got zero matches.
- **Count** of donations matching a filter (e.g. a date range) is available server-side without pagination via `stats[total]=count`, and the docs give a donations-specific example of exactly this. **Sum of amounts is not documented anywhere** — only `count` is a documented stat operation, so "Funds Raised" would require paginating through matching donations and summing `amount_in_cents` client-side.
- Each donation is **one row per individual charge/payment**, not one row per recurring plan. The schema has no field grouping repeat charges from the same recurring gift (no `recurring_donation_id`, no `payment_profile_id`, nothing) — this contradicts a secondary-source snippet I found, which I flag explicitly below.
- Given the schema (`signup_id` is a to-one FK on each donation row, and Signup itself carries a `donations_count` rollup confirming one signup can have many donations), **"Number of Donations" as raw row count = transaction count, not unique-donor count.** Transaction count is efficiently computable (`stats[total]=count`); unique-donor count is not natively supported and would need client-side de-duplication on `signup_id` across a full pagination pass. Which one the org actually *wants* is a policy call, not an API fact (see §5).

## 1. The donations endpoint(s)

The v2 REST resource is:

- `GET /api/v2/donations` — list/index, paginated (`operationId: listDonations`)
- `GET /api/v2/donations/{id}` — show a single donation
- `POST /api/v2/donations` — create
- `PATCH /api/v2/donations/{id}` — update
- `DELETE /api/v2/donations/{id}` — delete

(OpenAPI spec, `paths."/api/v2/donations"` and `paths."/api/v2/donations/{id}"`.)

I confirmed this is the **complete** list of fundraising-adjacent v2 resources by grepping every top-level path in the spec (`grep '^  "/api/v2/'`): the only donation/fundraising-related resources that exist anywhere in the v2 API are `donation_tracking_codes`, `donations`, and `pledges`. There is no `/donations/summary`, `/donations/stats`, `/recurring_donations`, `/donation_plans`, or `/payouts` path anywhere in the spec.

There *was* a legacy v1 Donations API (`GET/POST /api/v1/donations`, `GET /api/v1/donations/search`, `PUT`/`DELETE`), documented at [nationbuilder.com/donations_api](https://nationbuilder.com/donations_api), which explicitly states "the Donations v1 API has a [v2 equivalent](https://nationbuilder.com/api/v2/reference#get-/api/v2/donations)" — i.e. v1 is legacy and the org should build against v2.

**Auth/scopes:** The spec applies a single global security scheme — `security: [BearerAuth: []]`, described as "Authentication using a bearer token (JWT) issued via OAuth" (OpenAPI spec, `components.securitySchemes.BearerAuth`). I found **no per-endpoint OAuth scope strings** anywhere in the spec (searched for "scope" — zero matches). The 401 response schema's description reads: "The access token is missing, expired, or not authorized to access this resource," with an example message "...The resource owner also may not have a permission level sufficient to grant access" — implying access is gated by the NationBuilder user/nation-level permission model (e.g. whether the token owner has Finances access in the control panel), not by a named, separately-grantable OAuth scope. This is not spelled out further in the docs I could access.

## 2. Server-side date filtering, and count/sum without full pagination

**Date-range filtering is supported**, via the generic JSON:API-style filter syntax the whole v2 API uses. Per the OpenAPI spec's top-level description: "Filtering uses an operator syntax: `filter[attribute]=value` for equality..., and `filter[attribute][operator]=value` for other comparisons. String attributes support `eq`, `not_eq`, `eql`, `not_eql`, `prefix`, `not_prefix`, `suffix`, `not_suffix`, `match`, and `not_match`; **numeric and date attributes support `eq`, `not_eq`, `gt`, `gte`, `lt`, and `lte`**." (OpenAPI spec, `info.description`.) The Core Concepts guide gives worked `gt`/`gte`/`lt`/`lte` examples against a numeric attribute — `filter[donations_amount_in_cents][gt]=500`, `[gte]=500`, `[lt]=1000`, `[lte]=1000` — on the `signups` endpoint. ([Core Concepts](https://support.nationbuilder.com/en/articles/9757369-nationbuilder-v2-api-core-concepts))

The donation resource itself exposes date-time attributes suitable for this: `created_at`, `succeeded_at`, `pending_at`, `failed_at`, `canceled_at`, `updated_at` (OpenAPI spec, `donation_read_write_attributes` / `donation_read_only_attributes`). So `GET /api/v2/donations?filter[succeeded_at][gte]=2025-01-01&filter[succeeded_at][lt]=2026-01-01` follows the documented general pattern. **Caveat:** I could not find a donations-specific worked example using these date fields in the fetched docs — every concrete `gte`/`lte` example I found in the HOWTOs uses `signups`/`donations_amount_in_cents`, not `donations`/`succeeded_at` directly. The field names and general operator behavior are documented; the exact combination should be smoke-tested against a live token before the dashboard is built on it.

**Count without pagination — yes, and documented specifically for donations.** The Core Concepts guide states: "You can get the count of any API resource by adding the `stats[total]=count` parameter to your request... For example, the following will return the count of all donations: `/api/v2/donations?stats[total]=count`... The count will be returned in the `meta` field of the response... in the following format: `{ meta: { stats: { total: { count: 402 } } } }`." ([Core Concepts](https://support.nationbuilder.com/en/articles/9757369-nationbuilder-v2-api-core-concepts)) This is a query-parameter modifier that should compose with `filter[...]` on the same request (a combined worked example with both `stats[total]=count` and a `filter[...]` clause together was not found in the fetched docs, so combine-and-verify against a live token).

**Sum of amounts — not documented; no aggregate/summary endpoint exists.** I explicitly checked whether `count` is the only stat operation: re-fetching the Core Concepts guide for any mention of `sum`, `avg`, `average`, or per-attribute stats returned **zero** matches beyond the single `stats[total]=count` example. The generic response envelope schema for any paginated list (`index_document.meta`) is described only as: "Non-standard information about the document, such as requested statistics. **Empty unless the endpoint has metadata to convey.**" (OpenAPI spec, `components.schemas.index_document`) — i.e. `meta` is not auto-populated with a running total/sum; it only carries what you explicitly ask for via `stats[...]`, and `count` is the only documented stat. **There is no `/donations/summary` or similar aggregate endpoint** (confirmed by the full path list in §1). Getting "Funds Raised" for a date range therefore requires paginating through every matching `donations` row (`page[size]` max 100, default 20 — OpenAPI spec `pagination_size` parameter) and summing `amount_in_cents` client-side.

**A secondary-source discrepancy worth flagging:** one non-primary search snippet I encountered along the way claimed paginated responses carry a `meta.total_count`/`meta.total_pages`/`meta.current_page` block by default. I could not confirm this in either the OpenAPI spec (`pagination_links` only documents `self`/`first`/`last`/`prev`/`next` link URLs, no count fields) or the Core Concepts HOWTO (which only documents `stats[total]=count` as the way to get a count). Treat that specific claim as unconfirmed/likely wrong; `stats[total]=count` is the properly documented mechanism.

## 3. How recurring donations are represented

**There is no `recurring_donations` (or `donation_plans`/`subscriptions`) resource in the v2 API.** I searched the entire OpenAPI spec text for "recurring", "subscription", "installment", "frequency", "interval", "auto_renew", "payment_profile", and "monthly plan" — the only hits were unrelated fields on other resources (e.g. a path-journey due-date interval). Zero hits for anything recurring-donation-related.

The donation resource's writable/read-only attributes (OpenAPI spec, `donation_read_write_attributes` + `donation_read_only_attributes`) are exactly: `actblue_order_number`, `amount_in_cents`, `canceled_at`, `check_number`, `custom_values`, `donation_tracking_code_id`, `email`, `employer`, `failed_at`, `first_name`, `is_corporate_contribution`, `is_private`, `last_name`, `mailing_id`, `membership_id`, `middle_name`, `note`, `occupation`, `page_id`, `payment_type_id`, `payment_type_name`, `pending_at`, `pledge_id`, `signup_id`, `succeeded_at`, plus read-only `amount`, `created_at`, `import_id`, `payout_id`, `status` (enum `succeeded`/`pending`/`failed`/`canceled`/null), `updated_at`. There is no field here that groups multiple charges from the same recurring gift.

The full relationship/sideload list for a donation (OpenAPI spec, `donation_sideload_values` enum) is: `author`, `donation_tracking_code`, `import`, `mailing`, `membership`, `page`, `payment_type`, `pledge`, `recruiter`, `signup` — again, no `recurring_donation` relationship. The [Relationships between Resources](https://support.nationbuilder.com/en/articles/9912495-relationships-between-resources-in-v2-api) guide's relationship table lists "Donation" → "Signup" explicitly as a **To-One** relationship (a donation belongs to exactly one signup), consistent with the reverse being to-many — corroborated by the `Signup` resource itself carrying rollup counters `donations_count` ("Number of donations made by the signup") and `donations_amount_in_cents` ("Sum of donations made by the signup") as read-only attributes (OpenAPI spec, signup schema) — i.e. one signup routinely has many donation rows.

There **is** a separate `pledges` resource (`GET/POST /api/v2/pledges`, etc.), but a pledge is a one-time promise/commitment amount, not a recurring-charge schedule: its schema (`pledge_read_write_attributes`/`pledge_read_only_attributes`) has `amount_in_cents` (amount pledged), `amount_left_in_cents` ("Amount donated by the signup minus the amount pledged"), and a `signup_id`. Donations can optionally link to a pledge via `pledge_id`, but a pledge is not itself a repeating-charge object.

**Outside the API**, the control panel does have a recurring-giving concept: the [Recurring donation processing](https://support.nationbuilder.com/en/articles/2351041-recurring-donation-processing) HOWTO describes recurring gifts as listed under "Finances > More > Monthly plans," with each charge tied to a `payment_profile_id` ("a unique identifier used to recognize a monthly donation or installment plan"). **This `payment_profile_id` concept does not appear anywhere in the v2 API OpenAPI spec** — it may be a control-panel-only construct, or exposed through some mechanism not captured in the spec I fetched. This should be verified against a live NationBuilder instance/token before assuming it's unavailable via the API.

**Discrepancy flag:** a secondary (non-primary) search result claimed "the `recurring_donation_id` should not be set when creating a new donation" as if it were a current v2 field. I could not find `recurring_donation_id` anywhere in the current v2 OpenAPI spec (zero matches for "recurring" at all, as noted above). The `payout_id` part of that same secondary snippet, however, **is** confirmed directly in the spec (`donation_read_only_attributes.payout_id`: "The ID of the payout in which this NationBuilder Payments donation was included"). Treat the `recurring_donation_id` claim as unconfirmed/likely stale (possibly v1-era or from an older API version) — do not rely on it.

**Conclusion for "per year" bucketing:** because each recurring charge is (as far as the documented schema shows) an independent `donations` row with its own `succeeded_at`/`amount_in_cents`, and there's no documented field correlating repeat charges from the same recurring gift, year-bucketing "Number of Donations" by transaction date is well-defined at the row level — but there is no API-native way to instead bucket "Number of *recurring gifts*" as a single unit spanning multiple years, since nothing ties the rows together.

## 4. Transaction count vs. unique-donor count

Based strictly on the schema found in §1–3:

- Each `donations` row has exactly one `signup_id` (a to-one relationship to the donor's Signup record — confirmed both by the attribute definition, "The signup that donated," and by the Relationships guide's explicit "Donation → Signup: To-One" table entry).
- A single signup can and does have many donation rows — confirmed by the Signup resource's own rollup attributes `donations_count` / `donations_amount_in_cents` ("Sum of donations made by the signup") existing as lifetime aggregates, which would be meaningless if a signup could only ever make one donation.
- There is no "unique donor" stat operator, no `distinct` clause, and no dedicated resource for "donors" separate from `signups` — a donor is just a signup that happens to have donation rows pointing at it.

Given that, raw row-count on `/api/v2/donations` (via `stats[total]=count`) is **inherently a transaction count**, not a donor count — it counts charges, and a donor who gives 12 times in a year (e.g. a monthly recurring donor, per §3) contributes 12 to that count, not 1. Getting a **unique-donor** count for a date range is not natively supported by any endpoint or stat parameter I found: it would require paginating through every matching `donations` row for the period, collecting `signup_id` per row, and de-duplicating client-side — exactly the "pull every record" cost the org is trying to avoid, and the opposite of what `stats[total]=count` gives you for free.

**Recommendation (reasoning, not asserting an org decision):** "transaction count" is the metric that is efficiently and natively supported by the documented API (one `stats[total]=count` call per year, filtered by date). "Unique donor count" is a legitimate alternative reading of "Number of Donations" but is not efficiently computable from this API as documented — it would negate the "don't want to pull every record" goal. If the org wants a donor-count metric, that's a real, separate metric ("Number of Donors") that should probably be named and tracked distinctly rather than conflated with "Number of Donations," precisely because the schema treats these as two different things (a `donations` row count vs. a distinct-`signup_id` count).

## 5. Open policy decisions

These are not resolvable from the API docs — they are choices for the org to make:

1. **Does a recurring gift's individual charges count as N donations or 1?** The API has no field grouping repeat charges from the same recurring plan (§3), so even if the org *wants* "1 per recurring plan," there's no documented API mechanism to compute that without an out-of-band correlation (e.g. matching `signup_id` + amount + a monthly cadence in `succeeded_at` — fragile and unconfirmed against real data) or manual reconciliation against the control panel's "Monthly plans" view, which isn't exposed via this API surface as documented.
2. **Transaction count vs. unique-donor count for "Number of Donations"?** (§4) — the schema makes these two different, computably-distinct things; picking one is a reporting-definition choice, not an API fact.
3. **Which donations count toward "Funds Raised" — only `status: succeeded`, or also `pending`/`failed`/`canceled`?** The `status` enum (`succeeded`/`pending`/`failed`/`canceled`/null) exists on every donation row, and the API returns all statuses by default unless filtered. Nothing in the docs prescribes which should count as "raised."
4. **Which date field defines the "year" a donation belongs to?** A donation carries at least `created_at`, `succeeded_at`, `pending_at`, `failed_at`, and `canceled_at` as distinct timestamps (§2–3) — e.g. a donation created in December but not settling (`succeeded_at`) until January could land in a different year depending on which field is chosen. The docs don't designate one of these as "the" reporting date.
5. **Do refunds/cancellations reduce a prior year's "Funds Raised"?** The schema has a `canceled_at` timestamp and a `canceled` status value, but no explicit refund-amount or partial-refund field was found in the spec. Whether a `canceled` donation should net against previously-counted "succeeded" totals (and in which year) is not addressed by the docs.
6. **Should pledge-linked donations be treated differently from walk-up donations?** A donation can optionally carry a `pledge_id` (§3); whether fulfilling a pre-existing pledge should be counted the same as a fresh donation is a policy call, not an API fact.
7. **Is the control-panel "Monthly plans" / `payment_profile_id` grouping actually reachable via the v2 API at all** (e.g. as an `extra_fields` value or an undocumented field), or is it truly control-panel-only as the OpenAPI spec's silence on it suggests? This needs live-instance verification against a real NationBuilder token before the org commits to a "one row per charge, no grouping" assumption for the dashboard.
